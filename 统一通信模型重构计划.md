# Wombat.Network 统一通信模型重构计划

## 1. 目标

把当前分散的 `TCP`、`UDP`、`WebSocket`，以及后续要补的串口，收敛到一套统一的通信内核上：

- 底层统一为“传输适配器 + 管道收发循环 + 协议编解码 + 会话生命周期”模型。
- 对外统一抽象连接、监听器、会话、消息编解码、心跳、背压、关闭语义。
- 目标框架保持 `netstandard2.0`。
- 源码层面允许使用当前编译器支持、且不突破 `netstandard2.0` API 边界的最新 C# 语法。
- 项目文件显式固定 `LangVersion=latest`，语言能力按最新稳定编译器特性使用。
- 在 `netstandard2.0` 前提下尽量使用 `Span<T>`、`Memory<T>`、`ReadOnlySequence<T>`、`ArrayPool<T>`、`System.IO.Pipelines`，减少复制和分配。
- 本次为彻底破坏性升级，不保留旧公开 API 兼容层，直接切换到新接口模型。

## 2. 项目现状

### 2.1 已有能力

- `TCP` 客户端/服务端：`Wombat.Network/Sockets/Tcp/...`
- `UDP` 客户端/服务端：`Wombat.Network/Sockets/Udp/...`
- `WebSocket` 客户端/服务端：`Wombat.Network/WebSockets/...`
- 现成帧编解码：`Wombat.Network/Framing/...`
- 已引入 `System.IO.Pipelines`，并且仓库里已经有一套 `Pipelines.Sockets.Unofficial` 风格实现：`Wombat.Network/Pipelines/...`

### 2.2 当前主要问题

#### 1. 三套通信栈重复实现过多

- `TcpSocketClient`、`TcpSocketSession`、`UdpSocketClient`、`UdpSocketSession`、`WebSocketClient`、`WebSocketSession` 都各自维护：
- 状态机
- 接收循环
- 发送逻辑
- 心跳
- 超时
- 关闭清理
- 日志

这类代码现在不是“协议差异”，而是“运输层重复”。

#### 2. 现有 `Pipelines` 内核没有真正成为统一底座

- `Wombat.Network/Pipelines/SocketConnection.cs` 已经接近可复用底层。
- 但 `TCP/UDP/WS` 主流程基本没有统一跑在这套抽象上。
- `PipelineSocketConnection.cs` 又实现了一套新的管道 socket 包装，和 `SocketConnection` 职责重叠。

结论：当前仓库里实际上存在两套方向相近的底层，容易继续分叉。

#### 3. 现有帧接口是 `byte[]` 中心模型，零拷贝空间受限

- 现有 `IFrameBuilder/IFrameEncoder/IFrameDecoder` 主要围绕 `byte[]`、`offset`、`count`。
- 很多解码器会新建 `byte[]` 再 `Array.Copy`。
- 这会让 `PipeReader` 的 `ReadOnlySequence<byte>` 优势浪费掉。

#### 4. `PipelineSocketConnection` 仍然存在额外复制

- 其接收路径把 `Memory<byte>` 又转成新的 `byte[]` 再交给 `SocketAsyncEventArgs`。
- 这不是零拷贝，只是多了一层管道。

#### 5. `WebSocket` 和普通流式协议没有拆出层次

- `WebSocket` 当前把：
- TCP/TLS 连接
- HTTP Upgrade 握手
- WS 帧编解码
- KeepAlive
- 业务模块路由

都揉在 `WebSocketSession` 一层里。

这会导致后续统一模型时很难只替换“传输层”。

#### 6. 串口目前没有实现

- 仓库中没有 `SerialPort` 实现。
- 这反而是好事：串口可以直接按新模型接入，不需要先兼容旧实现。

## 3. 重构原则

### 3.1 先统一传输模型，不先统一业务 API

先把“如何收发字节流/消息”统一，再基于统一内核定义一套新的规范接口。这里不再考虑旧 API 迁移成本。

### 3.2 只保留一套内核

建议保留并演进 `Wombat.Network/Pipelines/SocketConnection.cs` 这条线，删除或冻结 `PipelineSocketConnection.cs`，不要维护两套 transport core。

原因很简单：

- `SocketConnection.cs` 更接近成熟的 duplex pipe 模型。
- 它已经围绕 `PipeReader/PipeWriter` 组织。
- 它比 `PipelineSocketConnection.cs` 更少“中间拷贝味道”。

### 3.3 传输层和协议层解耦

统一后必须分成四层：

1. 传输适配器层：TCP、UDP、串口、内存流
2. 管道层：统一 `PipeReader/PipeWriter`
3. 协议层：LengthField、LineBased、Raw、WebSocketFrame、用户自定义
4. 会话/宿主层：客户端、服务端、路由、事件分发、重连、心跳

### 3.4 `netstandard2.0` 下的零拷贝要现实一点

可以做到“大幅减少拷贝”，但不能承诺“全链路绝对零拷贝”。

可做的：

- `PipeReader.ReadAsync` + `ReadOnlySequence<byte>`
- `SequenceReader<byte>`
- `Span<T>` / `ReadOnlySpan<T>`
- `Memory<T>` / `ReadOnlyMemory<T>`
- `ArrayPool<byte>.Shared`
- 尽量让解码返回“切片视图”，不是新数组

做不到完全避免的：

- `netstandard2.0` 某些 API 不如新 runtime 直接
- `UdpClient` 本身就偏向返回新数组
- `SerialPort` 的 API 比 socket 更老
- `SslStream`、WS 握手、文本解码有些路径仍会分配

结论：目标应该是“关键热路径最少复制、最少装箱、最少 Task 噪音”，不是追求口号式零拷贝。

## 4. 目标架构

### 4.1 统一核心抽象

建议新增以下最小抽象。

当前已确定的第一版接口形状如下：

```csharp
public interface ITransportConnection
{
    string Id { get; }
    EndPoint LocalEndPoint { get; }
    EndPoint RemoteEndPoint { get; }
    IDuplexPipe Transport { get; }
    Task StartAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

public interface ITransportListener
{
    Task StartAsync(CancellationToken cancellationToken = default);
    Task<ITransportConnection> AcceptAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

public readonly struct ReceivedMessage
{
    public ReadOnlySequence<byte> Payload { get; }
    public EndPoint RemoteEndPoint { get; }
}

public interface IMessageChannel
{
    string Id { get; }
    ValueTask SendAsync(ReadOnlyMemory<byte> message, CancellationToken cancellationToken = default);
    ValueTask<ReceivedMessage?> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

public interface IMessagePipe
{
    ValueTask WriteAsync(
        PipeWriter writer,
        ReadOnlyMemory<byte> payload,
        CancellationToken cancellationToken = default);

    bool TryRead(ref ReadOnlySequence<byte> buffer, out ReadOnlySequence<byte> payload);
}
```

说明：

- `ITransportConnection` 只负责“字节怎么进出”。
- `ITransportListener` 只负责监听和接入连接。
- `IMessagePipe` 只负责“消息边界怎么切”。
- `IMessageChannel` 才是给上层使用的统一消息收发模型。
- 第一版刻意没有引入 `IAsyncDisposable`、`IAsyncEnumerable` 等额外接口，先保证 `netstandard2.0` 下最小依赖、最小实现面。

### 4.2 统一传输类型

建议拆成两类：

#### 流式传输

- TCP
- TLS over TCP
- WebSocket 的底层 TCP/TLS
- 串口
- 命名管道/匿名管道（以后要扩展也顺手）

统一特征：

- 有序
- 连续字节流
- 需要 framing 才能切消息

#### 数据报传输

- UDP

统一特征：

- 每次收到的就是一个 datagram
- 保留远端地址
- 上层可以选择“按包即消息”，也可以再套一层 framing

### 4.3 推荐的内部目录

```text
Wombat.Network/
  Transports/
    Abstractions/
    Tcp/
    Udp/
    Serial/
    Tls/
  Channels/
    StreamMessageChannel.cs
    DatagramMessageChannel.cs
  Protocols/
    Framing/
    WebSocket/
  Hosting/
    Servers/
    Clients/
```

直接按新目录和新接口迁移，不再为旧 API 预留转调层。

## 5. 各协议如何套入统一模型

### 5.1 TCP

最简单，第一批迁移。

方案：

- 监听端接入统一 `SocketConnection`。
- 每个连接得到一个 `IDuplexPipe`。
- 上层用 `StreamMessageChannel + IFrameBuilderV2` 读取消息。
- 心跳、超时、背压不再写在 `TcpSocketClient/Session` 里，改成通用连接中间件。

### 5.2 UDP

不要硬把 UDP 假装成 TCP。

建议：

- 新增 `IUdpTransport`，读出 `(ReadOnlyMemory<byte> datagram, EndPoint remote)`。
- 服务端维护 `UdpPeerSession`，但 session 只是“远端地址 + 元数据 + 发送入口”，不要伪造字节流。
- 默认直接按 datagram 交付。
- 如果用户配置了 `IMessagePipe`，再在 datagram 内做二次 framing。

一句话：UDP 统一的是“会话与消息模型”，不是“底层一定变成流”。

### 5.3 WebSocket

WebSocket 分两段：

1. 握手前：HTTP Upgrade
2. 握手后：WebSocket frame stream

建议：

- TCP/TLS 连接仍走统一 `ITransportConnection`
- 握手器单独作为 `WebSocketHandshakeMiddleware`
- 握手成功后，把底层 `IDuplexPipe` 包一层 `WebSocketMessageChannel`
- 上层只看到 text/binary/close/ping/pong 事件

重点不是把 WebSocket 当“特殊客户端类”，而是把它视为“建立在流式 transport 之上的协议层”。

### 5.4 串口

串口按流式传输接入，和 TCP 一样处理。

建议：

- 基于 `System.IO.Ports.SerialPort.BaseStream`
- 通过 `StreamConnection.GetDuplex(stream, ...)` 直接转成 `IDuplexPipe`
- 然后完全复用 `StreamMessageChannel`
- 继续使用现有长度帧、行帧、固定长帧

这会是统一模型的红利最大的一块，因为串口天然适合“流 + framing”。

## 6. 编解码接口必须升级

现有 `IFrameBuilder` 太偏 `byte[]`，建议直接升级为新的 `Pipe`/`Sequence` 风格接口：

```csharp
public interface IMessagePipeDecoder
{
    bool TryDecode(ref ReadOnlySequence<byte> buffer, out ReadOnlySequence<byte> payload);
}

public interface IMessagePipeEncoder
{
    ValueTask WriteAsync(PipeWriter writer, ReadOnlyMemory<byte> payload, CancellationToken cancellationToken = default);
}
```

### 6.1 解码原则

- 不返回新的 `byte[]`
- 返回 `ReadOnlySequence<byte>` 切片
- 由上层决定要不要 materialize

### 6.2 编码原则

- 直接往 `PipeWriter` 写
- 能预估长度的协议直接一次性 `GetSpan/Advance`
- 大消息用多段写入，不强行拼成一个数组

### 6.3 升级策略

- 直接废弃现有 `byte[]` 中心编解码接口
- 所有 framing 改为 `PipeWriter` 写入、`ReadOnlySequence<byte>` 解码
- 热路径协议优先改写，不做双栈共存

## 7. 性能策略

### 7.1 收包路径

流式协议统一：

- `Socket.ReceiveAsync` -> `PipeWriter.GetMemory` 直接填充
- `PipeReader.ReadAsync` -> `ReadOnlySequence<byte>` -> `SequenceReader<byte>` 解码
- 解码成功后只移动消费游标，不复制 payload

### 7.2 发包路径

- 业务写入 `Channel<OutboundMessage>`
- 单写者发送循环从 channel 取消息
- 编码直接写 `PipeWriter`
- transport 从 `PipeReader` 刷到底层 socket/stream

不要每次发送都临时建 `Task.Run`、`SemaphoreSlim`、`byte[]` 拼包。

### 7.3 缓冲管理

- 首选 `Pipe` 自己的分段内存管理
- 需要中间缓冲时才用 `ArrayPool<byte>.Shared`
- 删除大量“手工搬运数组”的代码路径
- 现有 `SegmentBufferManager` 可以保留做兼容，但不要继续扩散到新内核

### 7.4 并发模型

- 每连接固定两个主循环：`receive loop`、`send loop`
- 业务处理和网络 I/O 解耦
- 单连接发送端保持顺序，避免并发写 socket
- 服务端用 session registry 管连接，但不要在热路径上锁全局字典

### 7.5 心跳和超时

不要每种协议自己维护一套 `Timer`。

改成统一能力：

- `IConnectionHeartbeatStrategy`
- `IIdleTimeoutStrategy`
- `IKeepAliveStrategy`

TCP、UDP、串口都挂同一策略接口；WebSocket 的 ping/pong 只是具体实现。

## 8. `netstandard2.0` 约束与建议

### 8.1 可以继续用的能力

- `System.IO.Pipelines`
- `System.Memory`
- `Span<T>` / `Memory<T>`
- `ValueTask`
- `ArrayPool<T>`
- `ReadOnlySequence<T>`
- `SequenceReader<T>`

### 8.2 设计上要注意

- `Span<T>` 是 `ref struct`，不能跨 `await`
- 异步边界上统一传 `Memory<T>` 或 `ReadOnlyMemory<T>`
- 同步解析阶段使用 `Span<T>` / `SequenceReader<T>`
- 不要把 `byte[]` 当公共协议抽象
- 可以使用较新的 C# 语法糖、模式匹配、文件作用域命名空间、`readonly struct`、更简洁的初始化写法等
- 但不能把“新语法”误当成“新运行时 API 可用”，所有设计仍以 `netstandard2.0` 可访问 API 为准

### 8.3 语言版本建议

建议：

- 项目继续编译到 `netstandard2.0`
- 语言版本显式设为 `latest`
- 所有新代码优先使用更规范、更短、更不易出错的现代 C# 写法

边界：

- 可以升级语法，不等于可以依赖只存在于新 TFM 的 BCL API
- 如果某个写法依赖新的 runtime/library 特性，而不是纯编译器特性，就不能默认采用
- 当前已经确认这条路线可编译通过

### 8.4 包依赖建议

当前 `Wombat.Network.csproj` 里 `System.IO.Pipelines` 是 `8.0.0-rc.1...`，这不适合作为长期底座。

建议：

- 换成稳定版依赖
- 明确补齐 `System.Memory` 依赖
- 如果要支持串口，再按需引入 `System.IO.Ports`

## 9. 三阶段执行计划

### 第一阶段：统一通信内核

目标：

- 明确 `SocketConnection.cs` 作为唯一流式 transport 内核
- 删除或冻结 `PipelineSocketConnection.cs`
- 定义并稳定新的核心接口：`ITransportConnection`、`ITransportListener`、`IMessageChannel`、`IMessagePipe`
- 建立新的目录结构：`Transports`、`Channels`、`Protocols`
- 打通最小可用的 TCP 通路，作为整套模型的基准实现

本阶段交付：

- `Transports/Abstractions`
- `Transports/Tcp`
- `Channels/StreamMessageChannel`
- `Protocols/Framing`
- 最小 TCP demo / benchmark / tests

验收标准：

- 新 TCP 通路可独立收发
- `netstandard2.0 + LangVersion=latest` 编译稳定
- 分配次数、吞吐、并发能力至少不差于旧 TCP 实现

### 第二阶段：迁移全部协议到统一模型

目标：

- UDP 按 datagram 模型重做，不伪装成流式连接
- WebSocket 拆成 `TCP/TLS transport + 握手 + WS message channel`
- 串口基于 `SerialPort.BaseStream` 接入统一流式 channel
- 心跳、超时、背压、关闭语义统一收口到公共能力

本阶段交付：

- `Transports/Udp`
- `Transports/Serial`
- `Transports/Tls`
- `Channels/DatagramMessageChannel`
- `Protocols/WebSocket`
- 公共 heartbeat / timeout / close 策略

验收标准：

- TCP、UDP、WebSocket、串口全部跑在新模型上
- UDP 保留 datagram 语义
- WebSocket 传输层与协议层彻底拆开
- 串口可复用流式 framing

### 第三阶段：删除旧实现并完成切换

目标：

- 删除旧 `TcpSocketClient/Server/Session`
- 删除旧 `UdpSocketClient/Server/Session`
- 删除旧 `WebSocketClient/Server/Session` 中已被新架构替代的实现
- 删除重复 heartbeat/timer/buffer/状态机逻辑
- 删除 `byte[] + Array.Copy` 为中心的旧热路径
- 收敛测试、基准、文档、示例到新接口

本阶段交付：

- 只保留新接口和新实现
- 更新后的 README / API 文档 / 示例
- 新基准数据和迁移后的测试集

验收标准：

- 仓库内不存在旧通信栈双实现
- 公开接口全部切换为新模型
- 性能基准、压力测试、半包粘包测试、断连测试全部对齐新实现

## 10. 建议的执行顺序

按三阶段执行：

1. 第一阶段先把内核和 TCP 基准实现做对
2. 第二阶段一次性迁移 UDP、WebSocket、串口
3. 第三阶段删除旧实现并统一对外接口

原因：

- TCP 最标准，最适合先验证管道模型
- UDP 不该照搬流式模型，得单独单列
- WebSocket 层次最多，放在统一内核稳定后再拆更稳
- 串口没有历史包袱，直接接入新模型最省事
- 破坏性升级不需要长期双栈，但也不该在第一天就边删边重写全部协议

## 14. 风险点

### 14.1 最大风险不是兼容，而是接口边界定义不干净

破坏性升级反而少了一层包袱，但前提是新接口必须一次定义清楚：

- transport 和 protocol 不能再耦合
- stream 和 datagram 不能再混淆
- session 和 message 语义必须分离

### 14.2 UDP 语义不能被“统一抽象”误伤

UDP 的包边界、无连接、可丢包特性必须保留。统一模型只能统一宿主接口，不能伪装成 TCP。

### 14.3 `netstandard2.0` 会限制最理想写法

不要为了追求最新 API 放弃目标框架。该做的是把热路径写对，不是把代码写成现代语法展览。

## 15. 最终建议

建议按下面的判断执行：

- 统一内核：做
- 统一所有公开 API：直接做
- 立即保留旧类兼容：不做
- 先迁 TCP：做
- UDP 按 datagram 模型单独统一：做
- 串口等新内核稳定后直接接入：做

一句话版本：

这次不是“在旧 API 外面套一层管道”，而是“直接用一套规范的新接口重建 transport/channel/protocol 三层，把 TCP、UDP、WebSocket、串口全部迁到统一通信模型上”。
